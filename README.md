# Equity-Committee-Voting

## S3 Image Storage

Collateral images are stored in a private S3 bucket. Keep **Block all public
access** enabled in AWS S3. The backend returns short-lived presigned URLs when
committee users need to view images, so objects do not need public access.

Recommended bucket names:

- Development: `equity-committee-voting-images-dev`
- Staging: `equity-committee-voting-images-staging`
- Production: `equity-committee-voting-images-prod`

For AWS S3, configure the backend environment like this:

```env
S3_BUCKET=equity-committee-voting-images-dev
S3_REGION=us-east-1
S3_ENDPOINT=
AWS_ACCESS_KEY=your_access_key
AWS_SECRET_KEY=your_secret_key
S3_PRESIGNED_URL_MINUTES=15
```

For local MinIO, keep the endpoint set:

```env
S3_BUCKET=equity-committee-voting-images-dev
S3_REGION=us-east-1
S3_ENDPOINT=http://localhost:9000
AWS_ACCESS_KEY=minioadmin
AWS_SECRET_KEY=minioadmin
```

The backend will create the configured bucket automatically when using an
S3-compatible endpoint such as MinIO. For real AWS, create the bucket in AWS and
grant the backend IAM user only the minimum permissions it needs. On startup,
the backend verifies the AWS bucket with `HeadBucket` and fails fast if the
bucket name, region, or credentials are wrong.

Minimal IAM policy shape for `equity-committee-voting-images-dev`:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:ListBucket"],
      "Resource": "arn:aws:s3:::equity-committee-voting-images-dev"
    },
    {
      "Effect": "Allow",
      "Action": ["s3:GetObject", "s3:PutObject", "s3:DeleteObject"],
      "Resource": "arn:aws:s3:::equity-committee-voting-images-dev/cases/*"
    }
  ]
}
```

Keep **Block all public access** enabled. The app reads images through presigned
URLs generated by the backend.

To smoke-test the configured bucket from this machine, put the AWS settings in
`.env` and run:

```bat
run-s3-smoke.bat
```

The smoke test verifies bucket access, writes a tiny object under
`cases/smoke-test/`, reads it back, and deletes it.

## Local Backend Startup

Create or update `.env` in the repository root with your local settings. This
file is ignored by git. Use `.env.example` as the safe template.

Start the backend with:

```bat
run-backend.bat
```

The script loads variables from `.env`, sets Java 21, and starts Spring Boot with
the configured `SPRING_PROFILES_ACTIVE` value.

## Firebase Cloud Messaging

The mobile app can register FCM device tokens with the backend. The backend keeps
the normal in-app WebSocket notification path active and sends FCM push messages
only when Firebase Admin credentials are configured.

For local development, leave FCM disabled:

```env
FCM_ENABLED=false
FCM_SERVICE_ACCOUNT_PATH=
```

For an environment that should send push notifications, download a Firebase
service account JSON file outside source control and set:

```env
FCM_ENABLED=true
FCM_SERVICE_ACCOUNT_PATH=C:\secure\equity-voting-firebase-service-account.json
```

Service account JSON files are ignored by git. Do not commit Firebase
credentials.

For the Flutter client, add the Firebase app configuration files generated by
the Firebase console:

- Android: `mobile/android/app/google-services.json`
- iOS: `mobile/ios/Runner/GoogleService-Info.plist`

These files are also ignored by git. Android applies the Google Services Gradle
plugin only when `google-services.json` exists, so local builds without Firebase
project files still work.
